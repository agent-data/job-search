# Spec: Claude Code as a Job-Search Operating System

## Context

Job hunting is fragmented, manual, and noisy: people scroll job boards, copy postings into
spreadsheets, eyeball fit, and hand-tweak resumes per role. This project turns **Claude Code
itself into the operating system for that workflow** — a small set of skills, a private file-based
data workspace, the `agent-data` CLI as the data layer, and a local schedule as the "cron." A user
defines their preferences once (via an interview or by supplying their own brief), and from then on
Claude Code regularly pulls fresh postings, **judges each one's relevance against those preferences
using model inference**, and surfaces a digest of the matches — plus on-demand resume comparison and
truthful resume tailoring.

**Outcome:** an open-source project that, within ~5 minutes of install, shows a user real job matches
judged against their stated preferences, and then keeps doing so on a local schedule with zero ongoing effort.

**Design philosophy (set by the user):** relevance is a **qualitative judgment**, not a number.
Claude reads the posting against the prose preferences brief and decides **relevant / not relevant**,
and if relevant, **weak / moderate / strong** — with reasoning. No category weights, no 0–100 scores,
no false precision. Users tune the system in **human terms** (how often to pull), never in credits.

This document is the **design spec**. On approval, the implementation plan is produced via the
writing-plans skill (each skill built/tested with skill-creator).

### Research that shaped this design (already done)
- **Data source reality:** the agent-data marketplace has exactly **one** job listing — a
  LinkedIn-backed "Job Postings API" (`f9a6ec16-0bfd-44d8-b3ee-073776745ee7`), exposing a free
  `status` route, `search-jobs`, and `get-posting`. So v1 is LinkedIn-only; the source sits behind an
  adapter seam but we do **not** build a plugin system (YAGNI).
- **API reality:** `search` and `docs` are free; only `call` is metered (the free tier covers regular
  daily searching for most users). The design keeps pulls **frugal by behavior** and handles any quota
  limit **reactively, in plain language** — users never reason about credits.
- **Headless ≠ interactive:** scheduled runs have no human in the loop, so interactive skills (setup,
  interview, resume) are cleanly separated from the headless run.
- **Building blocks:** existing `job-preference-interview.md`; the `skill-creator` skill anatomy
  (SKILL.md + scripts/references, progressive disclosure, append-only JSONL state); and the `references/`
  DX guides (zero-friction T0, magical moment, name every error, opinionated defaults + escape hatches,
  four data paths, docs-as-product).

### Locked decisions (from the user)
1. **Local-first scheduling** — runs on the user's machine (OS cron / launchd, or `/loop`), using the
   already-authed `agent-data` CLI and local files. No cloud secrets. (Do **not** use `/schedule`, which
   creates cloud routines.)
   > **Revision (2026-06-08):** narrowed to **native `/loop` only** — Job Search OS no longer writes the
   > user's machine (cron/launchd generation removed). `/schedule` stays excluded (a cloud routine can't
   > reach the local workspace or `agent-data` auth). Current flow:
   > [`shared/references/internals.md`](../../shared/references/internals.md).
2. **Dedicated workspace folder** — private data lives in a folder the user picks (default `~/job-search/`),
   separate from the public code/plugin.
3. **User controls frequency, not budget** — default daily; users choose how often to pull (hourly,
   every few hours, daily, weekly). No credit budgets, no USD estimates, no per-call math as a user decision.
4. **Relevance is qualitative** — a dedicated evaluation skill judges **relevant vs not** (binary) and
   **weak / moderate / strong**. No weights, no numeric scores.
5. **Both packaging modes** — an installable Claude Code plugin **and** a repo of loose skills copyable
   into `~/.claude/skills/`.

## Goals & Non-Goals

**Goals (v1)**
- Define preferences via interview **or** import the user's own brief (interview is skippable).
- On a user-chosen schedule, pull postings via `agent-data`, dedup them, and **judge each for relevance**
  against the brief, producing a digest grouped by strong / moderate / weak match.
- Evaluate any single posting (pasted / saved / URL) against the brief on demand.
- Compare a resume against a posting (qualitative fit, strengths, gaps) — read-only.
- Tailor a resume to a posting **truthfully** (provenance-enforced, never overwrites the master).
- Be safely open source: no PII in the public repo; one-command install; honest, plain-language docs.

**Non-Goals / deferred (YAGNI)**
- Multi-source aggregation / a source-plugin system (design the seam only).
- A dedicated pipeline/triage board skill (the `jobs.jsonl` `status` field + digest cover v1).
- Numeric scoring, category weights, salary parsing (`salary_display` stays opaque text).
- Credit budgeting / cost dashboards (frequency is the only lever; quota limits handled reactively).
- Cloud sync, hosted dashboard, email/Slack notifications (local digest + optional desktop notify).
- Scraping application URLs (the API deliberately does not expose `application_url`).

## The Data Workspace

Default location `~/job-search/` (user-selectable). Everything here is personal/PII and must never
land in the public repo.

```
job-search/
  config.yaml                 # queries, schedule frequency, paths (NO budgets, NO score thresholds)
  preferences.md              # Job Preferences Brief — human prose only (must-haves, preferences, red flags)
  resumes/
    master.md                 # base resume(s); read-only to the tailor flow
    tailored/                 # generated tailored resumes (never overwrite master)
  jobs.jsonl                  # append-only EVENT log; current state = fold by source_id
  runs/<run_id>.json          # per-run audit log (status, results, errors)
  reports/<date>-digest.md    # human-readable digest per run
  .gitignore                  # deny-all template (workspace is private)
```

### `config.yaml` (human-terms only)
```yaml
version: 1
workspace:
  preferences_path: "preferences.md"
  master_resume_path: "resumes/master.md"
queries:                       # each enabled query = one search-jobs call per run
  - { id: "ai-eng-remote",  keywords: "AI engineer",          location: "United States",         limit: 25, enabled: true }
  - { id: "ml-platform-sf", keywords: "ML platform engineer", location: "San Francisco Bay Area", limit: 25, enabled: true }
schedule:
  frequency: "daily"           # hourly | every-2-hours | every-6-hours | daily | weekly
  time: "08:00"                # used for daily/weekly
  timezone: "America/Los_Angeles"
notify:
  digest_path_template: "reports/{date}-digest.md"
  desktop_notify_on_block: true   # one-time desktop ping if a scheduled run is blocked (auth/quota/no-brief)
```
No `budget`, no `scoring` thresholds — those are deliberately gone. `limit` is a search-breadth option
with a sane default, not a cost knob.

### `jobs.jsonl` — append-only **event** log (resolves append-only-vs-mutate)
Current state = fold events by `source_id` (last-write-wins per field). A `scripts/state.py` does the fold.
```jsonc
// written by the run when a NEW posting is found + evaluated:
{ "event":"evaluated", "ts":"…", "run_id":"…", "source":"linkedin",
  "source_id":"3901234567",            // THE DEDUP KEY (LinkedIn-native, stable across searches)
  "query_id":"ai-eng-remote", "title":"Senior AI Engineer", "company_name":"Acme",
  "location_display":"Remote (US)", "salary_display":"$180K–$220K",   // free text — never parsed
  "posted_at":"2026-06-02T…", "source_url":"https://www.linkedin.com/jobs/view/3901234567",
  "posting_id_at_seen":"jp_4f2a9c1d8b3e",  // jp_ id from search row: TRANSIENT pairing token, NOT a key
  "detail_read":true,                  // did we read the full description?
  "relevant":true,                     // BINARY judgment
  "match":"strong",                    // "strong" | "moderate" | "weak"  (null if not relevant)
  "reasoning":"Hits must-have remote+senior IC and the LLM stack; comp not stated.",
  "dealbreakers_hit":[],               // any must-have/dealbreaker observably violated
  "unknowns":["compensation not stated"],   // brief criteria the posting didn't address
  "needs_human_check":false,           // true when a dealbreaker can't be confirmed from the posting
  "status":"new", "first_seen":"…" }
// later human status change:
{ "event":"status_changed", "ts":"…", "source_id":"3901234567", "status":"interested", "note":"applied via referral" }
```
- **Dedup key is `source_id`, always.** The `jp_` id is listing-scoped and unstable across searches —
  used only as a short-lived pairing token to re-read detail, stored as `posting_id_at_seen`.
- **Idempotent re-runs:** before evaluating, fold to `Set<source_id>`; known IDs are skipped (no
  re-evaluation, no extra detail read). Re-running the same day yields no duplicate entries.

### Run log — `runs/<run_id>.json`
Records `status_probe`, per-query results/errors, an `errors[]` array (with `request_id`, `retryable`,
attempts), and `results_summary` (`total_results`, `new_postings`, `evaluated`, `detail_read`,
`relevant`, `strong`/`moderate`/`weak` counts). **No budget block, no credit/USD fields.**

### Preferences brief — `preferences.md` (human prose only)
The brief is the qualitative output of the interview (or the user's own doc): a short summary, **must-haves /
dealbreakers** (the binary filters), **strong preferences**, **nice-to-haves**, and **red flags**, each as
plain observable language ("remote-US or SF Bay," "ships fast, low process," "no on-call rotation"). It is
**prose the model reads and reasons over** — there is no machine-readable contract block, no weights, no
numeric rubric. The interview's job is to make these criteria concrete enough that a reader could judge a
posting against them. `created_at` is recorded so a stale brief can be gently flagged.

### Privacy / `.gitignore` (deny-all)
The workspace is career-sensitive PII. Default stance: **the workspace is not a git repo.** Setup scaffolds
a deny-all `.gitignore` and **refuses (or loudly warns) to scaffold inside a directory whose `.git` points
at a public remote.**
```gitignore
# workspace is PRIVATE
*
!.gitignore
!config.example.yaml
```

## The Skills

Six skills + a shared core. Only `job-search-run` and `evaluate-job-fit` are model-invocable (so a
schedule, or a "check for new jobs" / "is this job a fit?" turn, can fire them); everything else is
user-only (`disable-model-invocation: true`) because it spends a metered call, writes files, or needs
interactive input.

| Skill | Invocation | Purpose |
|---|---|---|
| `job-search-setup` | user-only | Front door: preflight CLI+auth, scaffold workspace, interview-or-import brief, choose **pull frequency** (plain-language nudge), run a first live sample search (the magical moment), print how to schedule. |
| `job-preference-interview` | user-only | Refactor of `job-preference-interview.md`: one-question-at-a-time interview → `preferences.md` prose brief (must-haves, preferences, red flags). **No numeric rubric.** |
| `job-search-run` | **user + model** | The scheduled entrypoint. Headless pull → dedup → **evaluate relevance** (following the evaluate-job-fit method) → read full descriptions for promising matches → `jobs.jsonl` + run log + digest. (See "The Headless Run Loop".) |
| `evaluate-job-fit` | user + model | **The relevance brain.** Reads one posting + the brief and judges, by model inference, **relevant / not relevant** and **weak / moderate / strong**, with reasoning, dealbreakers, and unknowns. Usable standalone (paste a JD) and followed by the run for batch evaluation. (See "Relevance Evaluation".) |
| `resume-compare` | user-only | Resume vs one posting → qualitative fit (weak/moderate/strong), matched strengths, gaps (labeled real/partial/unknown), talking points. Read-only on the resume. |
| `resume-tailor` | user-only | Truthfully tailor the master resume to a posting → new file in `resumes/tailored/` + provenance diff. Never overwrites master. (See "Resume Features".) |

**Shared core (NOT skills):** `shared/references/{agent-data-contract.md, errors.md, conventions.md}`
and `scripts/{state.py (fold/dedup), provenance.py (resume gate), build.sh (references sync),
schedule_hint.sh}`. Deterministic mechanics live here; **relevance judgment is model inference, owned by
the `evaluate-job-fit` skill** (prose method, no scripts, no numbers).

**Skill `description:` fields** are written WHAT + WHEN (e.g. `job-search-run`: "…run the scheduled
search, check for new jobs, or when invoked by a schedule"; `evaluate-job-fit`: "…judge whether a job
posting matches the user's preferences"). Descriptions tuned via skill-creator's trigger-eval loop.

## The Headless Run Loop (`job-search-run`) — most important section

**Principle: free gates before metered calls.** Free `status` probe → free dedup → judge relevance from
the free search summary → read the full description only for postings that look relevant. Branch retries
on the API's `retryable` boolean. **No silent failures:** a run that found nothing because LinkedIn was
down looks different in the digest from one where there were genuinely no new jobs.

### Algorithm
```
0. PREFLIGHT (free)
   a. Resolve workspace; missing config.yaml -> E-NO-CONFIG, HALT.
   b. agent-data whoami; api_key_set=false -> E-NO-AUTH, HALT.
   c. Load config; unknown major version -> E-CONFIG-VERSION, HALT.
   d. Load preferences.md; missing/empty -> PATH B (no-preferences), HALT gracefully (spend nothing).
   e. agent-data call <listing> status (free): "ok" proceed; "degraded" set flag + lower ambition;
      unreachable -> E-SERVICE-DOWN, write "service down" digest, HALT.
1. SEARCH (one metered call per enabled query):
   agent-data call <listing> search-jobs --keywords … [--location …] --limit …
     --fields id,source_id,source_url,title,company_name,location_display,salary_display,posted_at,detail_available
   502 search_failed (retryable) -> backoff retry; give-up -> record + continue. 2 consecutive 502s ->
     E-UPSTREAM-STRETCH, stop searching. 422/400 -> E-BAD-QUERY (name the bad param), continue.
   QUOTA error (out of free/plan calls) -> E-QUOTA (plain-language: pull less often / upgrade), HALT.
2. DEDUP (free): known=Set(source_id) from jobs.jsonl; new = results with non-null source_id not in known.
3. EVALUATE-FROM-SUMMARY (free; model inference via evaluate-job-fit method):
   for each new posting, judge from the summary fields: clearly NOT relevant (a must-have is plainly
   violated) -> record irrelevant, no detail read. Relevant or uncertain -> queue to read the full JD.
4. READ DETAIL for promising matches (one metered call each), most-promising first for run responsiveness:
   agent-data call <listing> get-posting --posting_id <jp_id> --source_url "<url>"
     400 invalid_pair (NOT retryable) -> stale pair; judge from summary only, note it. 502 (retryable) ->
     backoff; give-up -> summary-only + note. Success -> re-judge with description_markdown + missing_fields[]
     (treat missing_fields as "not stated", never as a negative). If many look relevant, read the top batch
     and mark the remainder "summary-only, not yet fully read" in the digest (freshness/latency, not budget).
5. PERSIST + REPORT: write 'evaluated' events to jobs.jsonl; write runs/<run_id>.json; write
   reports/<date>-digest.md grouped strong → moderate → weak, then a collapsed "filtered out (not relevant): N";
   print a short terminal summary.
```

### Retry / backoff (only for `retryable:true`)
Up to 3 attempts, exponential backoff with jitter (~1s/3s/7s). `degraded` status → fewer detail attempts.
**2 consecutive search 502s → stop searching this run** (LinkedIn stretch-outage). **Never retry**
`invalid_pair` / `invalid_request` / `unsupported_field` / quota (caller-side or hard limits).

### The four data paths — what the user SEES
1. **Happy:** digest header `9 new postings · 3 strong · 2 moderate · 1 weak · 3 filtered out.` then groups
   (title/company/location/match/one-line reasoning/url), strong first; a collapsed "not relevant" count.
2. **No preferences yet:** spends nothing; named cause+fix pointing to `/job-preference-interview` or the
   import path. **Headless run exits non-zero + optional one-time desktop notify** so a schedule doesn't
   silently no-op forever.
3. **Zero results — two sub-cases:** (3a) results returned but all already known → reassuring "No NEW
   postings — all N already in your database (normal between LinkedIn batches)." (3b) literally zero rows →
   actionable "broaden keywords / check `agent-data … status`."
4. **API failure:** (4a) **auth** → named fix (`export AGENT_DATA_API_KEY=…`, verify `whoami`), exit non-zero.
   (4b) **502** → retried; recovered → footnote; stretch-outage → "LinkedIn unreachable, will retry next run"
   with partial-result accounting. (4c) **invalid_pair** → expected staleness footnote, not an error.
   (4d) **quota** → see "Frequency & API usage" (plain-language: reduce frequency / upgrade; no credit math).

### Run health
The digest carries a mandatory **Run health** line: `healthy | partial (N errors) | degraded (LinkedIn flaky)
| blocked (action needed)`, plus an errors table with `request_id`s. A run is never silently "fine" when it wasn't.

## Frequency & API usage (no budgets, no credit-think)

The only lever the user touches is **how often to pull** (`schedule.frequency`: hourly → weekly). The system
keeps usage modest **by behavior**, not by a budget the user configures:
- It evaluates relevance from the **free** search summary first and reads a full description only for
  postings that look relevant — so metered calls track "promising matches," not raw volume.
- It dedups, so re-running never re-reads known postings.
- For run responsiveness it reads the most promising full descriptions first; any overflow is marked
  "summary-only, not yet fully read" (a freshness note, not a cost decision).

**If the agent-data API ever reports its quota is reached**, the run does not silently fail — it surfaces a
plain-language `E-QUOTA`:
> "agent-data's API limit for this period has been reached, so no new postings were pulled. This usually
> means searches are running very often. Lower the pull frequency in `config.yaml` (e.g. `daily` instead of
> `hourly`), or upgrade your agent-data plan at agent-data.motie.dev. Your existing matches are unaffected."

Setup gives a one-line qualitative nudge when choosing frequency ("Daily suits most searches; choose hourly
only if you're in a fast-moving, active search") — **no per-call math, no credit estimates** anywhere the
user makes a decision.

## Relevance Evaluation (`evaluate-job-fit`) — qualitative, by design

This skill is the relevance brain, and the `job-search-run` loop follows its method for batch evaluation
(loaded as guidance, applied across the run's postings — not re-invoked per posting). It is also directly
user-invocable to judge a single pasted/saved posting.

**Method (model inference, no numbers):**
1. Read the prose brief: must-haves/dealbreakers, strong preferences, nice-to-haves, red flags.
2. Read the posting (summary fields, or the full `description_markdown` when available).
3. Judge:
   - **Dealbreaker / must-have clearly violated → `relevant:false`** (reject), naming what failed.
   - **A must-have can't be confirmed from the posting → don't auto-reject;** keep it but set
     `needs_human_check:true` with the exact open question (e.g. "remote not stated — confirm before applying").
   - **Otherwise `relevant:true`**, with a coarse **`match`**: `strong` (hits the must-haves and most strong
     preferences), `moderate` (solid alignment, some gaps), `weak` (relevant but thin alignment).
   - Criteria the posting doesn't address → `unknowns` ("not stated"), **never counted against it**.
4. Output qualitative **reasoning** citing the posting against the brief.

**Output object:** `{ relevant, match: strong|moderate|weak|null, reasoning, dealbreakers_hit:[],
unknowns:[], needs_human_check }`. **No weights, no 0–100, no per-category numbers.** Consistency comes from
the skill's written method (judge dealbreakers first, then alignment; cite evidence) and from storing the
reasoning so a human can audit *why* something was called a strong match. The digest leads with the band and
the one-line reason — the reasoning carries the weight, not a number.

**Two depths, framed qualitatively:** judge from the summary to filter the obviously-irrelevant for free;
confirm and enrich the rest by reading the full description. `salary_display` is free text and is never
parsed for arithmetic; absence of any criterion is an unknown, not a fail.

**Determinism split:** dedup/state fold and the resume provenance check are **scripts**; **all relevance
judgment is model inference** owned by this skill.

## Resume Features

### `resume-compare` (read-only, safe)
- **Inputs:** resume path (default master) + one target posting (saved `source_id` / pasted JD /
  `source_url`+`posting_id` to read fresh). **Never writes the resume.**
- **Output** (terminal + optional `reports/compare-<company>-<date>.md`): a qualitative **fit
  (weak/moderate/strong)** with reasoning; MATCHED STRENGTHS (resume evidence ↔ JD requirement); GAPS labeled
  **real / partial / unknown** (don't conflate "you lack this" with "the JD didn't say"); TALKING POINTS; UNKNOWNS.

### `resume-tailor` (truthful; provenance-enforced)
Output is a **new** file `resumes/tailored/<company>-<role>-<date>.md` (master untouched) plus a tailoring diff.
- **Allowed (same facts):** reorder/re-emphasize bullets toward the JD; surface real keywords the candidate
  genuinely has, phrased to mirror JD vocabulary **only if true**; tighten language/formatting.
- **Forbidden (hard stops):** invent any experience/skill/employer/metric; inflate titles or change dates/
  company names; add a skill the master doesn't evidence (even if the JD demands it).
- **Enforcement (mechanical):** `scripts/provenance.py` requires every line in the tailored resume to trace
  to a master fact and emits `tailoring-report.md` (changed bullet → source bullet → transformation type).
  Anything lacking master provenance is surfaced in a top "⚠ REVIEW — no source in master" block and
  **excluded by default**; the human approves before the file is final. JD requirements the master can't
  satisfy go to a separate "Gaps you may want to address (NOT added)" section.
- **Ordering note:** `resume-compare` (zero write-risk) ships first; `resume-tailor` reuses its gap analysis
  and adds the provenance gate. Both are in v1 scope.

## Scheduling UX (`job-search-setup` makes the scheduled run trivial)

> **Revision (2026-06-08):** This section is the original cron/launchd design. Scheduling is now **native
> `/loop` only** — nothing is written to the user's machine. The current flow lives in
> [`shared/references/internals.md`](../../shared/references/internals.md); the block below is retained as a
> historical snapshot.

"Boring by default" = **OS cron / launchd invoking headless `claude`** (survives session-close and reboots).
`/loop` is the "I leave Claude open anyway" option. Setup translates the chosen `schedule.frequency` into the
right schedule string, does the magical first run, then **prints** copy-paste-ready scheduling:
```
OPTION A — OS cron (recommended; runs even when Claude is closed)
  crontab -e  →  0 8 * * *  cd ~/job-search && claude -p "/job-search-run" >> ~/job-search/runs/cron.log 2>&1
       (an hourly frequency would generate `0 * * * *`, etc. — setup writes the line matching your choice)
  • Verify now:  cd ~/job-search && claude -p "/job-search-run"
  • macOS: the Mac must be awake at run time — keep it on, use `caffeinate`, or install the launchd plist
    (StartCalendarInterval can wake the machine — the robust mac option).
OPTION B — keep Claude open and loop:  /loop <frequency> /job-search-run
Not sure? Use Option A.
```
- **Catch-up semantics:** the run is stateless — it always asks "what's new vs the DB's known `source_id`s,"
  so a missed run just means a slightly larger next batch. No backfill; setup says so.
- **Progressive consent:** setup may offer to write the crontab line itself (after showing it + a yes),
  writing a marker so it never re-asks.

## Open-Source Packaging & DX

**Dual distribution from one folder set.** The plugin and the loose-skills repo are the same `skills/`
directory referenced two ways. Because a copied-in skill folder must be self-contained, `shared/references/*`
is the source of truth and a tiny **`scripts/build.sh` fans it into each skill's `references/`** for the
loose-skills mode; the plugin references `shared/` directly. Authors edit one copy; CI syncs.

```
job-search-os/                 # PUBLIC repo (never any personal data)
  README.md  plugin.json  LICENSE (MIT or Apache-2.0)  CONTRIBUTING.md
  skills/{job-search-setup,job-preference-interview,job-search-run,evaluate-job-fit,resume-compare,resume-tailor}/
         SKILL.md [+ scripts/ + references/ (synced)]
  shared/references/{agent-data-contract,errors,conventions}.md
  scripts/{state.py, provenance.py, build.sh, schedule_hint.sh}
  templates/{config.example.yaml, preferences.example.md, workspace.gitignore}
  examples/sample-digest.md
```

**README / onboarding (chef-for-chefs):** one-line what + the magical moment up front; the single golden
TTFV path (`npm i -g agent-data` → set key / `agent-data whoami` → `/job-search-setup`, which ends by showing
real matches); the six skills with a real sample digest; the scheduling block; a **privacy** section
(deny-all `.gitignore`, never commit the workspace); a plain-language note that frequent pulls use the API
more and how `E-QUOTA` reads if a limit is hit; a **troubleshooting** named-error table (every `E-*` with
cause+fix). No credit math in user-facing docs.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| LinkedIn 502 stretches | Free `status` circuit-breaker; retry only `retryable:true`; stop after 2 consecutive search 502s; "will retry next run" digest; automatic catch-up. |
| Only one data source | Source behind `agent-data-contract.md` + a small search adapter (seam only — no plugin system in v1). |
| Very frequent pulls hit the API limit | Frugal-by-behavior (detail only for relevant); dedup; **reactive plain-language `E-QUOTA`** (reduce frequency / upgrade) — never a budget the user tunes. |
| Stale `jp_`/`source_url` pair | Never key on `jp_`; treat `invalid_pair` as expected staleness → summary-only judgment + footnote. |
| Machine off at schedule time | Stateless catch-up (no backfill); document `caffeinate`/launchd-wake; forgiving default time. |
| Brief drift | Store brief `created_at`; if old, digest nudges "re-run /job-preference-interview." Never auto-edit. |
| Judgment consistency | Qualitative by design (binary relevant + coarse band avoids false precision); consistency via the evaluate-job-fit written method (dealbreakers first, evidence-cited) + stored reasoning for audit. |
| Resume fabrication | `provenance.py` mechanically excludes non-master content; human approves; gaps shown as honest guidance. |
| PII leak via git | Deny-all workspace `.gitignore`; setup refuses/warns inside a public-remote repo; README privacy section. |

## Verification (end-to-end)

- **Preflight & auth:** with no key, `/job-search-run` prints the named `E-NO-AUTH` fix and spends nothing;
  `agent-data whoami` confirms once fixed.
- **Free probes only:** `agent-data call <listing> status` and `search` are free — exercise discovery and the
  circuit-breaker without metered spend; `--dry-run` verifies resolved requests.
- **Four data paths (+ quota):** feed the run canned `agent-data` JSON (happy / no-preferences / zero-results
  3a&3b / 502 / auth-fail / invalid_pair / quota) and assert digest wording, exit codes, and the run-health line.
- **Dedup idempotency:** run twice on the same canned results → no duplicate `evaluated` events, no extra detail read.
- **Relevance evals (skill-creator, qualitative):** a labeled set of postings + a sample prose brief → assert
  correct **relevant/not** calls, sensible **weak/moderate/strong** banding, dealbreaker rejection, and
  unknown handling ("not stated" never counted against a posting).
- **Resume guardrails:** master + a JD demanding an absent skill → assert the skill is reported as a GAP, never
  added; `provenance.py` rejects any unsourced line; master file is byte-identical after tailoring.
- **First-run UX:** time `/job-search-setup`; confirm it ends by displaying real relevance-judged matches within
  the TTFV target and prints copy-paste scheduling for the chosen frequency.
- **Packaging:** install as a plugin AND copy `skills/*` into `~/.claude/skills/`; confirm both load and
  `build.sh` keeps references in sync.
