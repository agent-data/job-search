# Hermes Harness Review — Overview

The entry point for the Hermes harness review: a maintainer scoping the [Hermes concierge layer](../2026-06-30-hermes-job-search-concierge.md) and its [execution plan](../../exec-plans/active/2026-06-30-hermes-concierge-layer.md) should read this page first, then drill into the six topic docs it links. The review's job is narrow — establish, from Hermes's own source, which harness mechanisms the concierge design leans on, and whether the shipped adapter ([`hermes.md`](../../../shared/references/platform/hermes.md), introduced by the [Hermes-native host design](../2026-06-29-hermes-native-plugin.md)) describes them correctly. It is not a design approval and it does not modify the adapter.

**How it was produced.** Each topic doc is grounded in a local clone of `NousResearch/hermes-agent` and source-cited line-by-line, then a separate fact-check pass re-read the clone to confirm or contradict the load-bearing claims. Hermes is not on CI, so nothing here is live-proven; the must-verify-live list at the foot of this page is the maintainer's bench checklist on a real install.

## Read commit and the divergence caveat — read this before trusting any source line

The clone that was actually read, and that the fact-check re-verified against, is **`184c10c`** (confirmed by `git rev-parse HEAD` on the on-disk tree); the files a real divergence would have removed — `tools/async_delegation.py`, `hermes_cli/subcommands/cron.py`, `docs/chronos-managed-cron-contract.md`, `tools/clarify_gateway.py` — are all **present** there. Disk and fact-check are the same commit, so the disk-vs-fact-check divergence this section originally warned about does **not** exist.

Because the codebase moved between these commits, several "X exists" / "X does not exist" assertions in the topic docs are **commit-sensitive and currently in conflict** — see the Corrections log. To keep this page's own links honest:

- **Corrections-log links point at `184c10c`** where a contested file exists only there.

## Corrections log

The fact-check pass contradicted nine line-anchored claims across four topic docs. The topic docs are **left as written** (re-pinning them to one commit is the proper fix, not patching individual lines against a commit not on disk); this log is the honest record of what a maintainer must not act on until reconciled. Items where the conflict only resolves at `184c10c` link that commit.

- **delegation-and-subagents.md — "delegate_task has no background/async path; it is inline/blocking; retire the adapter's background PIN."** Contradicted. At `184c10c`, top-level delegations run in the **background by default** (the model does not choose), returning a handle immediately with each result re-entering as a later message; [`tools/async_delegation.py`](https://github.com/NousResearch/hermes-agent/blob/184c10c/tools/async_delegation.py) exists and backs it. The adapter's background PIN was **correct**; the doc's advice to retire it and assert inline/blocking is actively harmful. *Status: open — keep the PIN; verify the execution model live.*
- **delegation-and-subagents.md — depth "clamped to [1,3]."** Contradicted. Default depth is 1 (flat), but there is **no upper ceiling** (range [1, ∞)); the doc is also internally inconsistent (it calls concurrency uncapped two rows above). *Status: open — low impact; depth stays 1 for the concierge regardless.*
- **cron-and-scheduling.md — "`--no-agent` is fabricated; delete it from the adapter."** Contradicted. `--no-agent` is a real registered CLI flag and a `cronjob()` schema property at `184c10c` ([`hermes_cli/subcommands/cron.py`](https://github.com/NousResearch/hermes-agent/blob/184c10c/hermes_cli/subcommands/cron.py)). *Status: open — do not scrub it from the adapter.*
- **cron-and-scheduling.md — "`hermes_cli/subcommands/cron.py` and `docs/chronos-managed-cron-contract.md` do not exist."** Contradicted. Both exist at `184c10c` ([chronos contract](https://github.com/NousResearch/hermes-agent/blob/184c10c/docs/chronos-managed-cron-contract.md)); the in-repo [Hermes-native host design](../2026-06-29-hermes-native-plugin.md) cites the same two files. *Status: open — adapter citations stand; the doc's verb-disambiguation table (CLI `edit` vs tool `update`) is still its most valuable, and confirmed, contribution.*
- **identity-architecture-and-install.md — "a no-origin cron job falls back to local (save to file), not a home channel."** Contradicted. The router iterates configured home channels and returns the first match, falling to local-only only when none is set; the home-channel-first behavior is real and matches the cron and tools/clarify siblings. *Status: open — the identity doc is the outlier.*
- **tools-clarify-and-channels.md — "the messaging gateway wires no clarify; the string `clarify` does not appear in `gateway/run.py`" (graded BLOCKING).** Contradicted. `clarify` appears many times at `184c10c` and a [`tools/clarify_gateway.py`](https://github.com/NousResearch/hermes-agent/blob/184c10c/tools/clarify_gateway.py) pending-session path exists. The BLOCKING severity is undermined; the residual question (interactive choices vs free-text reply on messaging) is a live check, not a known breakage. *Status: open — downgrade from blocking to a verify-live caution.*
- **tools-clarify-and-channels.md — "WhatsApp has no home-channel fallback; it always needs an explicit chat_id."** Contradicted. A `WHATSAPP_HOME_CHANNEL` mapping exists. *Status: open — low impact.*
- **skills-system-and-packaging.md — the clarify auto-append cited at `tools/clarify_tool.py:19`.** Conclusion confirmed (auto-append + duplicate-on-authored-Other), but the citation points at a comment; the real append is in the CLI renderer, as the tools/clarify sibling states. *Status: open — fix the source attribution; the T4 fix it supports is sound.*
- **Adjusted in this stitch pass (not a Hermes finding).** The six topic docs were split across two directories (`hermes-harness-review/` and a divergent `2026-06-30-hermes-harness-review/`); they are now consolidated into `hermes-harness-review/`, the index links repointed, and this `overview.md` added, so `python3 scripts/doc_lint.py --root .` passes index-completeness and internal-links. *Status: done.*

## Consolidated implications for the concierge layer

Cross-cutting findings, most-severe first. Each is grounded in a Hermes-source fact and tied to a concierge claim; the per-claim detail is in the linked topic doc.

### Blocking

- **Host-gating has no concrete detection mechanism.** The design dedicates a section to "Hermes-only behavior is host-gated" and the exec plan's non-goal is "No cross-host behavior change," but neither names how `job-search` learns it is on Hermes. Hermes exposes no host==Hermes boolean; `HERMES_PLATFORM`/`HERMES_SESSION_PLATFORM` are the messaging channel (telegram/discord/cli/…), not host identity, and the repo's existing multi-harness model already selects "the active adapter" by model judgment rather than a pinned signal. Misfiring the gate means either cross-host bleed (violating the non-goal) or Hermes falling back to the portable flow. **Decide before building**: pin a deterministic signal (presence of `~/.hermes/`, a set `HERMES_HOME`, or `hermes` on PATH, or a Hermes-only bundled marker) and wire exec T3 Step 2 to it. *(identity-architecture-and-install.md)*

### Cautions

- **Calibration assumes synchronous detail reads; top-level `delegate_task` may run in the background.** Design §7→§8 reads a batch, summarizes, then asks for the user's reaction as one flow; at `184c10c` a top-level delegation returns immediately and results arrive as later messages. This is commit-sensitive (the on-disk `f98b5d0` may differ) and the adapter already PINs it — so keep the PIN, verify the model live, and treat the **sequential in-turn read** as the safe path for a calibration batch that must be assembled before the reaction ask. *(delegation-and-subagents.md)*
- **Install-by-copy silently enrolls job-search in the default-ON Curator.** A pack copied into `~/.hermes/skills` can be marked stale, archived, or LLM-consolidated between runs — and a job-search that idles between scheduled runs is exactly the "unused" profile the Curator prunes. The design treats copy-in as an equal option. Prefer tap/hub or `skills.external_dirs` (not Curator-eligible); the adapter Packaging section and design Architecture should name the trade-off. *(skills-system-and-packaging.md)*
- **The memory permission gate cannot keep memory out of context, and Hermes writes `USER.md` autonomously by default.** `MEMORY.md`/`USER.md` is auto-injected before any ask, so §3/§4 should read "may be used to *draft* only with permission," not "may *inform* … only with permission"; and with `write_approval` off by default, an inferred preference can become durable profile truth without the user confirming durability — contradicting the design's "don't silently promote" policy. Keep inferred prefs in the workspace brief only, never write `USER.md` without explicit confirmation, and consider setting `memory.write_approval=true` as a hard gate. *(memory-and-sessions.md)*
- **"Deliver here" and "create the job immediately" depend on the in-session `cronjob` tool and a running gateway.** Origin binds to the session only at create time, so the verbatim shelled `hermes cron create … --deliver origin` from a bare terminal degrades to a home channel or local-only; scheduled delivery fires only while the gateway runs. The concierge should create recurring jobs via the tool and confirm the gateway before promising automation (§11/§12). *(cron-and-scheduling.md, tools-clarify-and-channels.md)*
- **Closed-choice menus may not render natively on messaging surfaces.** Messaging clarify exists but its render path is unverified; design §3/§10/§11 menus and the §5 interview should degrade to numbered prose where clarify is not interactive, rather than assume a choice picker everywhere. The exec T4 fix (open-ended ⇒ pass no choices ⇒ free-text) is correct, but the eval should assert that behavioral contract, not that a "clarify bug" was patched. *(tools-clarify-and-channels.md)*
- **§11 conflates a new platform with a new chat.** Standing up a brand-new platform requires an out-of-band provider/token + gateway setup, and the gateway restart tears down the active session, so "hand off and resume in one continuous flow" is not achievable for that case — only for a new chat on an already-configured platform. Split the two and reframe new-platform setup as "configure out-of-band, then resume on the next interaction." *(tools-clarify-and-channels.md)*
- **The bundled-runtime invocation rides on a template token, and the headless exit code is unproven.** `${HERMES_SKILL_DIR}` is substituted into the rendered `SKILL.md` at load (gated by `skills.template_vars`), not exported to the shell, so the runtime call must resolve its directory from the injected skill path / run path and tolerate the token being disabled. Separately, the `hermes chat -Q` exit code and whether `hermes -z` honors `-s` are unproven — keep the written run record as the primary signal for any cron/headless wrapper. *(identity-architecture-and-install.md, skills-system-and-packaging.md)*

### Meta

- **Re-pin before editing the adapter.** The commit divergence above is the root cause of the contradicted claims. Pick the one commit the maintainer's Hermes runs, re-verify every existence claim and PIN against it, and only then use these docs to edit `hermes.md`. *(this overview, Read-commit caveat)*

## Strengths to keep

The fact-check confirmed the concierge's core bets at the harness layer: the Variant-A split (canonical skills + a `hermes/` source layer bundled into them, backstopped by build-then-byte-equality validation); confining memory/session drafting to the interactive front door (cron is `skip_memory` with no auto-injected context); treating the local workspace brief — not Hermes's tiny, autonomously-curated global memory — as canonical state; the clarify open-ended/closed-choice discipline; keeping §7 calibration and §10 automation distinct so calibration never routes through cron; and the explainability model (stored artifacts first, optional live re-check), which maps cleanly onto Hermes's own cron status, last-status/last-error, and suppressed-but-saved output signals.

## must-verify-live

On a real Hermes from this branch, settle: (1) the exact commit the maintainer's Hermes runs, and re-verify every existence claim against it; (2) the top-level `delegate_task` execution model — inline/blocking vs background/async — which decides whether calibration results are in-turn; (3) how `clarify` renders on a messaging surface (interactive choices vs a pending free-text reply); (4) the `hermes chat -Q` exit-code semantics and whether `hermes -z` honors `-s/--skills`; (5) that `${HERMES_SKILL_DIR}` resolves in the runtime call (and the fallback when `skills.template_vars` is off); (6) whether the agent's `terminal` subprocess inherits the session origin so `--deliver origin` targets this chat; (7) that `hermes skills install` (not just `tap add`) is required to load the skill, and that the bootstrap/verify docs include it; (8) that copy-into-`~/.hermes/skills` is Curator-eligible while tap/hub and `skills.external_dirs` are not.
